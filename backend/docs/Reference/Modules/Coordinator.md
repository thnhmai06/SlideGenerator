# Coordinator Module

The **SlideGenerator.Coordinator** module manages system-wide concurrency and resource throttling.

## Responsibility
- Preventing resource exhaustion (CPU/RAM).
- Managing per-gate semaphore limits.

## GateLocker System
The `IGateLocker` service provides a "gate-based" locking mechanism.

### Gate Types
- `DownloadImage`: Throttles HTTP requests to cloud providers.
- `EditImage`: Limits concurrent MagickImage processing (heavy CPU/RAM).
- `EditPresentation`: Ensures only one thread modifies an output PowerPoint at a time.
- `ReadWorkbook` / `ReadPresentation`: Throttles I/O for large document parsing.

### Dynamic Limits
Limits are read from `SlideGeneratorSettings` at runtime. If the user changes the "Max Parallel Downloads" in the UI, the `GateLocker` respects the new limit on the next acquisition.

## Asset Deduplication
The `ICoordinator` service handles deduplication of computationally expensive or bandwidth-heavy assets within a single workflow.

### Use Case
If a spreadsheet has 100 rows pointing to the same high-resolution product image with identical crop/resize parameters, the system should only perform the download and Magick.NET processing once.

### Mechanism
- **Task Coordination**: Multiple steps requesting the same asset (calculated via a composite key of URI, dimensions, and edit options) join a shared `Task`. 
- **Primary vs. Secondary**: The first step to request the asset becomes the **Primary** (performing the actual work). Subsequent requests become **Secondary** (waiting for the Primary task).
- **Hard-Linking**: Instead of copying bytes or re-editing, the Secondary step creates a **Hard Link** (via `SlideGenerator.Utilities/HardLink`) to the file generated by the Primary.

### Scope
Coordinators are workflow-scoped and transient. They do not persist state between different generation requests, ensuring that changes to source assets are eventually reflected in new workflows.

## Implementation (GateLocker)
Uses a `ConcurrentDictionary` of `GateState` objects, each containing a `LinkedList` of `TaskCompletionSource` waiters to ensure O(1) removal and fair FIFO scheduling.
